# 웨네버 광고 중심 수익 모델

## 결론

웨네버는 구독 모델이 맞지 않다. 학생이 이 앱에 돈을 내는 이유는 약하고, 성적 분석도 “AI에게 내 공부를 맡긴다”가 아니라 `내가 지금 몇 등쯤인지`, `상위 10% 안에 드는지`, `내 점수가 어느 위치인지`를 잠깐 확인하는 용도에 가깝다.

따라서 수익 모델은 `무료 사용 + 광고 + 앱 실행/촬영/분석/결과 진입 광고 + 시험기간 직접 스폰서십`으로 가는 게 맞다.

핵심은 광고를 많이 넣는 것이 아니라, 학생이 이미 기다리는 순간과 화면 하단 여백에 자연스럽게 넣는 것이다.

## 수익 구조

```text
월 매출 =
  배너 광고 노출 수 / 1000 * 배너 eCPM
  + 앱 실행/결과 진입 광고 노출 수 / 1000 * 전면 eCPM
  + 촬영/분석 화면 네이티브 광고 노출 수 / 1000 * 네이티브 eCPM
  + 석차/분포 추가 확인 리워드 광고 노출 수 / 1000 * 리워드 eCPM
  + 시험기간 직접 스폰서십
```

구독은 제외한다. 유료 기능을 고민한다면 나중에 `광고 제거 1회성 구매` 정도만 검토하고, MVP 수익 모델에는 넣지 않는다.

## 광고 위치

### 1. 앱 실행 광고

앱을 켤 때 노출하는 광고다. 단, 매번 띄우면 바로 이탈하므로 `콜드 스타트 + 빈도 제한`이 필요하다.

권장 방식:

- 첫 홈 화면 진입 직전 또는 직후에 전면/팝업 광고
- 사용자당 하루 1회, 많아도 2회 이하
- 로그인/학생증 인증이 끝난 사용자에게만 노출
- 닫기 버튼 명확히 제공
- 시험기간에는 직접 스폰서 캠페인으로 교체 가능

피해야 할 것:

- 앱 실행 때마다 무조건 전면광고
- 학생증 인증 전 광고
- 오류/로딩 화면 위 광고
- 광고 닫기 버튼을 작게 만들거나 늦게 보여주는 방식

### 2. 사진 촬영/자르기 화면 광고

사용자가 사진을 찍고 자르는 동안 하단에 네이티브 광고를 넣는다. 첨부한 예시처럼 작업 화면은 유지하고, 하단에 광고 카드가 들어가는 방식이 좋다.

적용 위치:

- 시간표 OCR 사진 선택/촬영 후 미리보기 화면
- 시험지/문제 사진 촬영 후 자르기 화면
- 학생증 인증 화면은 제외

권장 방식:

- 하단 네이티브 광고 카드 1개
- `AD` 라벨 명확히 표시
- 사진 영역, 자르기 핸들, 확인 버튼을 가리지 않음
- 광고 영역은 접힘/고정 높이로 레이아웃 흔들림 방지

이 위치는 전면광고보다 낫다. 사용자가 이미 사진 확인에 몇 초를 쓰기 때문에 광고가 UX를 덜 해친다.

### 3. AI 분석/석차 계산 대기 광고

가장 자연스러운 광고 지점이다. 사용자는 어차피 결과 계산을 기다리므로, 이 시간에 스폰서 카드나 네이티브 광고를 보여준다.

사용자 흐름:

1. 시험 선택
2. 내 점수 입력 또는 사진 업로드
3. `내 위치 확인하기` 탭
4. 3-5초 분석/계산 대기 화면
5. 결과 화면 노출

광고 적용:

- 분석 대기 화면에 스폰서 카드 1개 노출
- 대기 화면 하단 또는 중앙에 네이티브 광고
- 하루 무료 확인 횟수 초과 시 `광고 보고 1회 더 보기`
- 직접 광고 상품은 이 위치를 최우선으로 판매

권장 문구:

- `잠시만요. 같은 시험 제보를 기준으로 위치를 계산하고 있어요.`
- `광고를 보면 오늘 1회 더 확인할 수 있어요.`
- `표본이 적으면 실제 석차와 다를 수 있어요.`

### 4. 결과 보기 직전/결과 화면 광고

결과 화면은 사용자가 가장 궁금해하는 순간이라 광고 단가가 높을 수 있다. 다만 너무 세게 막으면 반감이 크다.

권장 방식:

- `결과 보기` 버튼을 누른 뒤 전면광고를 하루 1회 이하로 테스트
- 전면광고 대신 `스폰서 혜택 팝업`을 결과 진입 전에 노출
- 결과 화면 하단에는 작은 배너만 허용
- 결과 수치와 그래프 사이에는 광고를 넣지 않음

좋은 흐름:

1. 계산 완료
2. `결과 보기` 버튼
3. 전면광고 또는 스폰서 팝업
4. 닫기
5. 석차/분포 결과 전체 노출

나쁜 흐름:

- 결과 일부만 보여주고 광고를 강제로 봐야 나머지 공개
- 석차 수치 위에 광고 팝업을 덮음
- 결과를 볼 때마다 전면광고

### 5. 하단 배너

하단 배너는 작게 깔아두는 기본 매출원이다. 단독으로 큰돈은 안 되지만 앱 전체 트래픽이 쌓이면 안정적인 바닥 매출을 만든다.

권장 위치:

- 홈 화면 하단
- 급식 화면 하단
- 시간표 화면 하단
- 게시판 목록 하단
- 성적 시험 목록 하단

피해야 할 위치:

- 글쓰기 입력창 바로 위
- 댓글 입력 중 키보드 위
- 학생증 인증 화면
- 계정 삭제/신고/운영 관련 화면

### 6. 게시판 네이티브 광고

게시판은 배너보다 네이티브 광고가 자연스럽다.

권장 방식:

- 게시글 8-12개마다 광고 카드 1개
- `광고` 라벨 명확히 표시
- 일반 게시글처럼 공감/댓글이 되는 형태는 금지
- 광고 신고 버튼 제공

초기에는 게시판 광고를 많이 넣지 말고, 학교별 DAU가 충분히 나온 뒤 테스트한다.

### 7. 시험기간 직접 스폰서십

프로그램매틱 광고보다 더 중요한 수익원이다. 학생 앱의 트래픽은 단가가 낮을 수 있어서, 시험기간에 직접 광고 상품을 팔아야 한다.

판매 상품:

| 상품 | 가격 예시 | 노출 위치 |
| --- | ---: | --- |
| 앱 실행 스폰서 | 300,000-1,000,000원/2주 | 앱 실행 직후 팝업 |
| 촬영 화면 네이티브 | 300,000-1,000,000원/2주 | 사진 촬영/자르기 하단 |
| 시험기간 스폰서 카드 | 500,000-2,000,000원/2주 | 석차 확인 대기 화면 |
| 결과 진입 스폰서 | 500,000-2,000,000원/2주 | 결과 보기 직전 |
| 학교/지역 배너 패키지 | 200,000-700,000원/월 | 홈/급식/시간표 하단 |
| 자료실 스폰서 | 500,000-2,000,000원/월 | 무료 자료, 이벤트, 특강 |
| 전국 시험기간 패키지 | 별도 협의 | 여러 학교 동시 노출 |

초기 영업 대상:

- 문구/플래너/노트 브랜드
- 독서실/스터디카페
- 지역 학원 단기 특강
- 교육 콘텐츠/문제집 브랜드
- 학생 행사/공모전/캠프

주의:

- 개인 성적을 근거로 특정 학원 광고를 보여주면 안 된다.
- 광고주는 학교/지역/시험기간 단위로만 타겟팅한다.
- `고3 수학 60점 이하` 같은 세분 타겟팅은 하지 않는다.

## 석차 기능의 포지셔닝

앱의 성적 기능은 `AI 분석`이 아니라 `익명 제보 기반 석차/분포 추정`으로 잡아야 한다.

메인 가치:

- 내 점수가 상위 몇 %쯤인지
- 현재 제보 기준 예상 등급이 어디인지
- 같은 시험에서 최고점/중앙값/상위권 컷이 어느 정도인지
- 표본이 늘면서 결과가 어떻게 바뀌는지

AI는 보조로만 쓴다:

- “표본이 적어서 신뢰도 낮음”
- “상위권 표본이 몰려 있을 수 있음”
- “현재 제보 기준 상위 10% 근처”

즉, 사용자가 원하는 건 긴 분석 리포트가 아니라 빠른 위치 확인이다.

## 광고 빈도 제한

학생 대상 앱이라 광고 피로도를 빡세게 관리해야 한다.

| 광고 슬롯 | 형식 | 제한 |
| --- | --- | --- |
| 앱 실행 | 전면/팝업 | 사용자당 하루 1회부터 시작 |
| 사진 촬영/자르기 | 하단 네이티브 카드 | 촬영 세션당 1개 |
| AI 분석/석차 계산 대기 | 스폰서 카드/네이티브 | 계산 요청마다 가능 |
| 결과 보기 직전 | 전면/스폰서 팝업 | 사용자당 하루 1회 이하 |
| 결과 화면 하단 | 작은 배너 | 화면당 1개 |
| 게시판 네이티브 | 피드 카드 | 게시글 8-12개마다 1개 |
| 추가 석차 확인 | 리워드 광고 | 사용자가 선택했을 때만 |

추천 MVP 설정:

- 1차: 촬영 화면 하단 네이티브, 분석 대기 스폰서 카드, 결과 화면 하단 배너
- 2차: 앱 실행 팝업 광고
- 3차: 결과 보기 직전 전면광고 A/B 테스트
- 4차: `광고 보고 1회 더 보기` 리워드 광고

우선순위는 `촬영 화면`, `분석 대기`, `결과 진입`이다. 이 세 곳이 사용자의 목적과 광고 노출이 가장 자연스럽게 맞는다.

## 매출 시나리오

프로그램매틱 광고만으로는 초기에 매출이 작다. 그래서 직접 스폰서십을 같이 붙여야 한다.

아래는 보수적 예시다. 실제 eCPM, fill rate, 광고 네트워크, 시즌성에 따라 크게 달라진다.

| 단계 | MAU | 광고 매출 | 직접 스폰서십 | 총매출 |
| --- | ---: | ---: | ---: | ---: |
| 초기 검증 | 5,000 | 10만-50만원 | 0-100만원 | 10만-150만원 |
| 학교 확장 | 30,000 | 80만-300만원 | 200만-700만원 | 280만-1,000만원 |
| 지역 확장 | 100,000 | 300만-1,200만원 | 800만-3,000만원 | 1,100만-4,200만원 |

해석:

- 작은 배너만으로는 큰돈이 안 된다.
- 석차 확인 플로우 광고가 일반 배너보다 단가가 좋다.
- 시험기간 직접 스폰서십이 수익의 핵심이 된다.
- MAU보다 `시험기간 석차 확인 횟수`가 더 중요한 지표다.

## MVP 광고 상품

### 무료 사용자

- 시간표/급식/게시판 무료
- 성적 석차 확인 하루 1회 무료
- 추가 확인은 광고 시청으로 획득
- 앱 실행 광고 노출
- 촬영/자르기 화면 하단 네이티브 광고 노출
- 분석 대기 스폰서 카드 노출
- 결과 보기 직전 광고 노출 가능
- 결과/홈/급식/시간표 하단 배너 노출

### 광고주 상품

- 학교/지역 단위 노출
- 시험기간 집중 노출
- 앱 실행, 촬영 화면, 분석 대기, 결과 진입 위치 중 선택
- 광고 성과 리포트 제공: 노출, 클릭, 신고, 숨김

## 데이터 모델 초안

```sql
create table public.ad_impressions (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references public.profiles(id) on delete set null,
  school_id uuid references public.schools(id) on delete set null,
  placement text not null,
  provider text not null,
  campaign_id text,
  created_at timestamptz not null default now()
);

create table public.score_check_usage (
  user_id uuid not null references public.profiles(id) on delete cascade,
  usage_date date not null,
  free_count int not null default 0,
  rewarded_count int not null default 0,
  primary key (user_id, usage_date)
);

create table public.sponsor_campaigns (
  id uuid primary key default gen_random_uuid(),
  title text not null,
  advertiser_name text not null,
  placement text not null,
  target_region text,
  target_school_id uuid references public.schools(id),
  starts_at timestamptz not null,
  ends_at timestamptz not null,
  status text not null default 'draft',
  created_at timestamptz not null default now()
);
```

- 학교 1-3곳에서 광고 피로도 확인
- 결과 보기 직전 전면광고 A/B 테스트
- 시험기간 스폰서 제안서 만들기
- 지역 광고주 20곳 직접 영업

## 정책/운영 원칙

- 개인화 광고는 기본 off로 둔다.
- 성적, 반, 시간표, 학생증 정보를 광고 타겟팅에 쓰지 않는다.
- 학교/지역 캠페인은 개인별 프로파일링이 아니라 학교 전체 또는 지역 전체 단위로만 운영한다.
- 광고에는 명확한 `광고` 라벨을 붙인다.
- 전면광고에는 닫기/스킵 버튼이 명확해야 한다.
- 부적절한 광고 신고 기능을 넣는다.
- 학생 대상이라 광고 카테고리를 보수적으로 차단한다.
